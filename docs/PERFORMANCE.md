# Производительность

Производительность — архитектурное свойство, а не поздняя оптимизация. Dual Frontier целится в конкретные цифры с первого дня; граф зависимостей, доменные шины и кэши с инвалидацией — прямые инструменты достижения этих целей.

## Целевые метрики

Сравнение с RimWorld показывает, какие именно операции должны быть дешевле и как это достигается.

| Операция                          | RimWorld             | Dual Frontier цель    | Как достигается                    |
|-----------------------------------|----------------------|-----------------------|------------------------------------|
| Поиск предмета на складе          | O(n) каждый тик      | O(1)                  | Кэш с инвалидацией                 |
| 100 пешек × патрон-запрос         | 6000 сканов/сек      | ≤100 сканов/сек       | Батчинг намерений                  |
| Параллельные системы              | 1 поток              | N-2 потоков           | Граф зависимостей                  |
| Загрузка мода                     | Перезапуск           | Горячая загрузка      | AssemblyLoadContext                |
| Нарушение изоляции                | Тихий баг            | Краш + диагностика    | SystemExecutionContext             |
| Тик систем настроения             | Каждый тик           | 1 раз/сек             | TickScheduler SLOW                 |

Цели формулируются пер-операции. Итоговый таргет фрейма — 16.6 мс при 60 FPS на сцене из 30 пешек и 10 000 тайлов; 33 мс при 30 FPS на сцене из 100 пешек. Эти значения фиксируются в CI через `PerformanceGate` — тесты падают, если регрессия превышает 10%.

## dotTrace и BenchmarkDotNet

### dotTrace

Основной инструмент для профилирования в игре. Используется в двух режимах:

- **Sampling** (`--profiling-mode=Sampling`) — низкий оверхед, подходит для полного runs на 60 секунд. Результат: где именно тратится CPU.
- **Timeline** — привязка событий шины и фаз планировщика к временной шкале. Для Dual Frontier это первичный режим: видно, какие фазы заканчиваются слишком поздно, какая шина держит поток, где `[Deferred]` очередь растёт.

Каждая фаза планировщика помечается через `Activity`/`EventSource`, доменная шина публикует `BusPublished` markers. dotTrace строит наглядную карту: "фаза 2, 8 мс, 6 систем параллельно, 2 простаивают".

### BenchmarkDotNet

Микробенчмарки для hot-путей: ComponentStore (Add/Remove/Query), DependencyGraph (BuildGraph), SpatialGrid (QueryRadius). Результаты хранятся в `tests/DualFrontier.Core.Benchmarks/Results`, сравниваются между версиями.

```csharp
[MemoryDiagnoser]
[SimpleJob(RuntimeMoniker.Net80)]
public class ComponentStoreBenchmark
{
    private ComponentStore<HealthComponent> _store = null!;
    [GlobalSetup] public void Setup() { /* ... */ }

    [Benchmark] public HealthComponent Get10k()
    {
        // типичный hot-путь.
    }
}
```

Бенчмарки запускаются в CI раз в неделю и ad-hoc локально при правке ядра.

## Горячие пути

Анализ нагрузки показывает следующие горячие пути — в порядке убывания частоты вызова.

### ComponentStore.Get / Query

Вызывается тысячи раз за фазу. Реализация на SparseSet: один массив для lookup, один для плотного хранения. Цель — 5 ns на `Get`, 30 ns на `Query<T1, T2>` per-entity.

### DomainEventBus.Publish

Публикация события синхронно обходит список подписчиков. Реализация — массив делегатов (не `List<T>`), без аллокаций. Цель — 50 ns на публикацию с 3 подписчиками.

### SpatialGrid.QueryRadius

Поиск entity в радиусе R от точки. Реализация — сетка клеток, каждая клетка хранит `List<EntityId>`. Цель — проход по ≤9 клеткам вместо всех 10 000 entity.

### SystemExecutionContext.GetComponent (DEBUG)

В DEBUG вся проверка декларации даёт +1-2 ns на вызов. В RELEASE проверка уходит.

### PathfindingService

A\* через навигационный граф. Самая дорогая операция (1-5 мс на путь). Кэш путей между часто используемыми парами точек даёт 10x.

## Кэши и инвалидация

Кэш без инвалидации хуже, чем отсутствие кэша: игра начинает врать. В Dual Frontier все кэши явно декларируют источник инвалидации.

### Кэш `InventorySystem`

Словарь `Dictionary<ItemType, List<EntityId>>`. Инвалидируется событиями `ItemAddedEvent` и `ItemRemovedEvent`. Проверка корректности — в DEBUG: раз в 100 тиков полный скан сверяется с кэшем.

### Кэш `PathfindingService`

Двухслойный: блокировки местности (НЕ меняется между стройкой) и путь между опорными точками. Инвалидация по `BuildingPlacedEvent`, `BuildingDestroyedEvent`, `TileChangedEvent`.

### Кэш `SpatialGrid`

Обновляется на каждый `PositionChangedEvent`. Не инвалидируется, а обновляется инкрементально — энтити удаляется из старой клетки, добавляется в новую.

### Кэш графа систем

`DependencyGraph` строится один раз и не меняется. Инвалидация только при загрузке/выгрузке мода — в этот момент граф пересобирается и планировщик пересчитывает фазы.

Правило: каждый кэш документируется таблицей `(ключ → значение, источники инвалидации, DEBUG-проверка)`. Без такой таблицы ревью кэша не проходит.

## Как добавить бенчмарк

1. **Определи, что измеряем.** Бенчмарк без цели — шум. Хорошая формулировка: "ComponentStore.Get on 10k entities должен держать ≤10 ns per call".
2. **Создай класс в `tests/DualFrontier.Core.Benchmarks`.**

    ```csharp
    [MemoryDiagnoser]
    [SimpleJob(RuntimeMoniker.Net80)]
    public class MyBenchmark
    {
        [Params(1000, 10000, 100000)] public int N;
        // ...

        [GlobalSetup] public void Setup() { /* ... */ }

        [Benchmark] public void HotPath() { /* ... */ }
    }
    ```

3. **Запусти локально.** `dotnet run -c Release --project tests/DualFrontier.Core.Benchmarks`.
4. **Зафиксируй baseline.** Результаты сохраняются в `Results/MyBenchmark.json`.
5. **Добавь в CI gate.** В `tests/DualFrontier.Core.Benchmarks/PerformanceGates.cs` добавь правило: "метод X должен быть не медленнее baseline +10%". CI падает при регрессии.
6. **Документируй.** Добавь строчку в таблицу "Горячие пути" с ожидаемой цифрой.

Бенчмарк без зафиксированного baseline не попадает в main — иначе любой может ухудшить производительность без шума.
